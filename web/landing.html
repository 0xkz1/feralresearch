Not an AI product dashboard. The interface treats automation as one layer inside a broader authored research and image-making practice.

Living system

The site should feel like a lab that keeps updating.

Instead of showing fake growth numbers, Feral shows the actual research objects moving through the system: disaster events, habitat features, prompt runs, browser research tasks, visual notes, and authored image studies.

Collecting

Disaster seeds

External source events are gathered server-side and stored as structured records before any interface renders them.

Mapping

Habitat context

Interpreting

Visual packets

Machine-generated prompts sit beside field notes, thumbnails, paintings, and editorial judgment.

How it works

A backend engine feeds a living research interface.

Disaster data enters server-side, habitat context is mapped, prompt packets are stored in SQLite, and the frontend reads everything through a local API.

Data ingestion

USGS earthquake feeds and GDACS RSS arrive through n8n workflows and are dispatched to the sidecar.

Habitat mapping

Overpass API queries farms, water, shelters, and infrastructure near the disaster epicentre.

Visual synthesis pipeline

Sidecar builds scenario JSON, generates prompts, and stores runs in SQLite for the frontend.

Authorship

Make the artist visible inside the system.

The interface should show that Feral is research-led visual practice. Generated material is framed as source material and interpretation, not as the whole work.
